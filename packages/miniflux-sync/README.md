# miniflux-sync

Idempotent sync of Miniflux assets and feed configuration from a JSON manifest.

```sh
miniflux-sync                       # apply ~/.config/miniflux/feeds.json
miniflux-sync sync path/to/feeds.json # custom manifest
miniflux-sync --dry-run             # report planned changes only
miniflux-sync bootstrap-user ...    # seed a user/API key in PostgreSQL
```

Read/star history is preserved:

- A feed is created only when its `feed_url` is not already on the server.
- The feed URL is an immutable identity key, not a managed mutable setting.
  Changing a declared URL creates a new feed and reports the old one as an
  orphan; migrate URLs manually in Miniflux first to preserve entry state.
- Setting changes go through `PUT /v1/feeds/<id>`, which never touches entries.
- Feeds are never deleted by the sync. To remove a feed, drop it from the
  manifest **and** delete it via the Miniflux UI.

Token sources, in order:

1. `MINIFLUX_TOKEN`
2. `MINIFLUX_TOKEN_COMMAND` (defaults to `rbw get miniflux-api-key`)

Feed settings support Miniflux API field names such as `title`, `site_url`,
`crawler`, `scraper_rules`, `rewrite_rules`, `disabled`, `hide_globally`,
`disable_http2`, `proxy_url`, and `user_agent`.
For feed secrets, use `cookie_file`, `username_file`, or `password_file`; their
contents are read at runtime and sent as `cookie`, `username`, or `password`.

`bootstrap-user` performs only the minimal DB writes required to authenticate the
sync: user row, default `All` category, integrations row, and API key. OIDC
subject mismatches fail hard.

Manifest examples are generated by `nixosModules/miniflux/default.nix`; malt's
source feed declarations live in `machines/malt/modules/miniflux/default.nix`.
